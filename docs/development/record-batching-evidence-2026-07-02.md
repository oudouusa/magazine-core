# Record Batching Evidence — 2026-07-02

## Scope

This slice closes the downstream-evidenced generic gap where plugins that
discover many records had to emit one JSON-RPC frame per `SourceRecord`.

The chosen design is SDK-level batching over the existing protocol v1 `record`
notification shape:

- `context.send_record(record)` remains unchanged and emits a single `record`.
- `context.send_records(records)` chunks to `MAX_RECORD_BATCH` and emits
  `records` batches.
- The Rust host continues to enforce `MAX_RECORD_BATCH`, `max_records`, and
  spool byte limits before ingest.

## Invariants

- `protocol_version = 1` and `record_schema_version = 1` remain unchanged.
- No private downstream names, real data, responses, cookies, proxies, or
  operational details are included.
- The Python SDK root imports remain compatible.
- The core repository has no dependency on downstream consumers.

## Synthetic Evidence

Added public-safe synthetic coverage for:

- SDK chunking into `100, 100, 5` record batches.
- Empty `send_records([])` producing no record frame.
- Python SDK `send_records()` through the Rust CLI/host ingest path.
- Host acceptance of `records` batch notifications.
- Host rejection of malformed notifications that specify both `record` and
  `records`.
- Host rejection of batches larger than `MAX_RECORD_BATCH`.
- Host `max_records` enforcement across a batch before spool.
- A pinned `record_batch` conformance fixture generated by the Python oracle.

## Decision

Adopt. The generic behavior is implemented with an additive SDK helper and a
documented protocol v1 batch fixture. No release tag is cut by this slice.

## Residual Risks

- `send_records()` requires a host that supports the protocol v1 `records`
  batch shape. This repository's host does; downstream lock bumps must verify
  their pinned host before retiring one-record-per-frame workarounds.
- Batch frame size is still bounded by the global 8 MiB protocol frame limit;
  plugins with very large records may need smaller application-level chunks.
